01

Overview

The system was built to do more than return a number. It persists a realistic graph in SQLite, computes shortest paths through two genuinely different execution strategies, validates every result against the underlying edge weights, and ships the whole thing behind an API and a visual frontend — so the algorithm's behaviour, not just its output, can be inspected.

Persistent graph database

5,000 nodes and tens of thousands of weighted edges stored in SQLite, with a dedicated table for caching computed paths.

Two solver engines

A sequential/parallel Python implementation alongside a native C++ solver that parallelises relaxation with OpenMP.

Built-in benchmarking

A full benchmark suite measures every configuration and exports human-readable and CSV reports automatically.

Architecture

Two routes lead from the same algorithm layer to the same result — one through the Python interpreter, one through compiled, multi-threaded C++. The interface and database don't know which route was taken.

04

Module breakdown

database.py
Graph persistence and retrieval
  • generate_sample_data() builds the 5,000-node graph
  • get_graph_as_dict() loads it as an adjacency list
  • save_path_result() caches computed paths
dijkstra.py
Sequential and parallel Python solvers
  • dijkstra_sequential() priority-queue baseline
  • dijkstra_parallel() multi-threaded variant
  • validate_path() checks results against edge weights
dijkstra_openmp.cpp
Native multi-core solver
  • Reads the SQLite graph directly
  • Parallelises neighbour relaxation with OpenMP pragmas
  • Compiled via build_openmp.sh / .bat
performance.py
Benchmarking and reporting
  • run_full_benchmark() across thread configurations
  • generate_performance_report() writes a .txt summary
  • export_results_csv() for external analysis
api.py
REST service layer
  • /setup · /stats · /shortest-path
  • /benchmark · /health
  • Built on FastAPI + Uvicorn
frontend/App.tsx
Interactive dashboard
  • Triggers setup and path queries against the API
  • Renders results with vis-network
  • Built with React 18 + Vite

Performance benchmark

20 randomised source/destination pairs were run against each configuration on the 5,000-node, 17,519-edge graph.

Sequential

average execution time per query, lower is faster

Why parallel is slower here: Python's Global Interpreter Lock prevents true concurrent bytecode execution, so thread/process management overhead outweighs any benefit at this graph size. The native dijkstra_openmp.cpp solver exists precisely to sidestep this limitation.
ConfigurationSpeedupEfficiencySuccess
Parallel · 1T0.36×35.91%18/20
Parallel · 2T0.16×8.23%20/20
Parallel · 4T0.08×1.93%19/20
Takeaway: for graphs of this size, the sequential implementation is the practical default. Parallel execution is reserved for the native OpenMP path or substantially larger workloads.
